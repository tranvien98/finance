## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `hsl(240 10% 3.9%)` — shadcn zinc-950 | Page background, auth page backdrop |
| Secondary (30%) | `hsl(240 3.7% 15.9%)` — shadcn zinc-900 | Auth card surface, settings card surfaces, expandable card interiors |
| Accent (10%) | `hsl(263 70% 60%)` — purple-500 equivalent | Reserved list below |
| Destructive | `hsl(0 72% 51%)` — red-600 | Destructive confirmation dialogs only |

Accent reserved for:
1. Primary submit buttons ("Sign in", "Create account", "Save API key")
2. Active tab indicator on the login/signup tab toggle
3. Focus ring on form inputs
4. Reveal-button icon when in "revealed" state (password/API key visible)

Not accent: inactive tabs, secondary text, card borders, helper text, disabled states.

Dark mode is the primary mode for Phase 1. Light mode tokens are not specified here — Phase 7 handles the mode toggle. All color values above reference dark mode CSS variables from shadcn's default slate/zinc dark theme.

Gradient background on auth page: `from-zinc-950 via-zinc-900 to-purple-950/20` (top to bottom), covering full viewport. This delivers the fintech-dark direction from D-02 without a custom palette.

## Screen Contracts

### Screen 1: Auth Page (`/auth` or `/login`)

**Layout:** Full-viewport centered column. Gradient background (`from-zinc-950 via-zinc-900 to-purple-950/20`). Single `Card` (400px wide on desktop, fluid on mobile) centered horizontally and vertically with `min-h-screen flex items-center justify-center`.

**Card contents (top to bottom):**
1. App name: "Finance" — Display typography (28px, weight 700)
2. Tagline: "Track expenses. Stay in control." — Body typography (14px, weight 400, muted text)
3. `Tabs` component: "Sign in" | "Create account" — active tab has accent underline indicator
4. Tab panel: form fields + submit button

**Sign in tab form:**
- Email field (type email, autocomplete email)
- Password field (type password, autocomplete current-password)
- "Sign in" button (accent, full width, height 44px)
- Inline error: red text (14px, weight 400) rendered below the relevant field — never as toast (D-04)

**Create account tab form:**
- Email field (type email, autocomplete email)
- Password field (type password, autocomplete new-password) — minimum 8 characters, no special character requirement
- Confirm password field (type password, autocomplete new-password)
- "Create account" button (accent, full width, height 44px)
- Inline error: same pattern as sign in

**Loading state during submit:** Button shows spinner icon (lucide `Loader2`, `animate-spin`) + label changes to "Signing in..." or "Creating account...". Button disabled. No skeleton needed — transition is sub-second.

**Redirect on success:** `/expenses` (D-01)

---

### Screen 2: Settings Page (`/settings`)

**Layout:** Standard authenticated page layout (sidebar nav + main content area — nav is scaffolded here for later phases, content area only for this phase). Main content: max-width 720px, centered, top padding 64px (3xl token).

**Three expandable cards** stacked vertically, 32px gap between them (xl token):

#### Card 1: API Keys
- Card title: "API Keys" (Heading, 20px, weight 600)
- Description: "Connect external AI services for expense classification." (Body, 14px, muted)
- Expandable: card shows collapsed summary, click chevron to expand
- Collapsed state: shows "OpenRouter API Key" label + masked value `sk-...xxxx` in a `Badge` (secondary variant) + "Configured" or "Not configured" status text
- Expanded state: Input field for API key + "Save API key" button (accent) + inline validation result

**API key save flow (D-07):**
1. User enters key and clicks "Save API key"
2. Button enters loading state: spinner + "Validating..."
3. On success: inline green text "API key validated and saved" below input
4. On failure: inline red text "Invalid API key — check your OpenRouter credentials" below input
5. On success only: input reverts to masked Badge display

**Reveal button (D-05):**
- When key is saved, show masked Badge + "Reveal" button (ghost variant, 14px)
- Clicking Reveal: replaces Badge with full key in Input (readonly), button changes to "Hide", lucide `Eye` → `EyeOff` icon. Accent icon color when revealed.
- Auto-hides after 30 seconds without user interaction

#### Card 2: Telegram Bot
- Card title: "Telegram Bot" (Heading, 20px, weight 600)
- Description: "Automatically log expenses by sending messages to your bot." (Body, 14px, muted)
- Expandable: same chevron pattern
- Collapsed: "Bot Token" label + configured/not-configured status
- Expanded: Input for Telegram Bot Token + "Save token" button (accent) + inline result
- Content is scaffolded here but full webhook logic is Phase 5

#### Card 3: Account
- Card title: "Account" (Heading, 20px, weight 600)
- Description: "Manage your account and session." (Body, 14px, muted)
- Expandable: same chevron pattern
- Collapsed: user email address
- Expanded: email display (readonly) + "Sign out" button (destructive variant, ghost style — red text, no background unless hovered)

**Settings page loading state:** While fetching existing key from DB, show `Skeleton` component (full card width, 80px height) for each card — 3 skeletons total.

## Animation Contract

| Interaction | Animation |
|-------------|-----------|
| Tab switch (Sign in ↔ Create account) | Content cross-fade, 150ms ease-out |
| Expandable card open | Height from 0 to auto, 200ms ease-out |
| Expandable card close | Height to 0, 150ms ease-in |
| Button loading state | Spinner `rotate` infinite, 600ms linear |
| Inline validation error appear | Opacity 0 → 1, translateY 4px → 0, 150ms |
| Reveal/hide key | Instant swap (no animation) |

Use Tailwind transition utilities. Do not use framer-motion — not in the stack.
